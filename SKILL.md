# 访谈小结结构化纪要

## 目标

把中文访谈材料整理成可进入调研报告的结构化小结，并默认交付为 Excel `.xlsx` 文件。内容形态不是散文式摘要，而是“研究维度 × 受访者”的小结矩阵；当用户只提供单个受访者时，也要按同一套维度组织，方便后续追加其他受访者。

始终只使用用户提供的附件、笔录、提纲、背景说明、参考表。不搜索网络，不补充行业常识，不用模型知识填空。

## 先读参考结构

开始写小结前，读取 `references/output-schema.md`，按其中的矩阵结构、单元格写法和质检规则执行。

如果用户提供已有小结 Excel 或模板，优先沿用该模板的 sheet、行标题、列标题、模块层级和语言颗粒度；不要擅自改成 Markdown 报告。

除非用户明确要求“只在聊天里展示/先给预览/不要生成文件”，否则最终必须生成 `.xlsx`。

## 文件读取预处理

收到用户文件后，先判断格式，再尝试读取：

- `.docx`、`.txt`、`.xlsx`：直接读取，进入整理流程。
- `.pdf`：尝试读取；若输出乱码、段落错位或说话人标签丢失，提示用户"该 PDF 读取质量不佳，建议用 WPS/Word 打开后另存为 .docx 再重新提交"。
- 扫描图片型 PDF：如果当前环境支持 OCR，则对扫描版 PDF 进行文字识别并回看原页核对；若不支持或识别质量不足，应提示用户提供可复制文本版 PDF、Word 或逐字稿。
- 其他格式（`.pages`、`.odt` 等）：尝试读取，失败则提示转为 `.docx`。
- 无文件读取工具的环境：提示用户直接将文字内容粘贴到对话中。

判断读取质量的标准：能否识别说话人标签（如"受访者："或时间戳）、段落是否连贯、是否存在大量乱码字符。

## 输入判断

先判断用户给了哪些材料：

- 访谈笔录：逐字稿、转写稿、访谈记录、会议记录。
- 访谈提纲：研究员的问题、模块、追问项。
- 项目信息：产品/功能介绍、调研背景、调研目的、目标用户。
- 参考小结：已有 Excel、Markdown、表格或历史小结样本。
- 多受访者材料：多个笔录、多个受访者列，或需要汇总到同一张表。

材料不足时仍可整理，但只能输出材料中能支撑的内容。缺失信息不要猜；必要时写“未提及”或留空，取决于用户是否要求保留固定表格。

## 核心工作流

1. 识别项目和场景
   - 从题名、提纲、开场白和高频问题中判断项目主题。
   - 识别是否需要拆 sheet，例如娱乐场景/学习场景、购买行为/制作行为、不同产品模式等。
   - 识别受访者身份、核心场景、常用产品和关键背景。

2. 建立小结 schema
   - 若有参考 Excel：复用参考表的行维度。
   - 若有访谈提纲：把问题组转换成分析维度，而不是逐题照抄。
   - 若只有笔录：从材料中归纳 6-12 个可横向比较的研究维度。
   - 每个维度要能服务后续报告分析，例如“使用场景”“全流程”“需求差异”“痛点”“产品机会点”“付费意愿”。

3. 做证据覆盖映射
   - 私下列出笔录中所有有效信息块：背景、场景、行为、工具、流程、需求、痛点、评价、机会点、付费态度、原话。
   - 给每个信息块分配到一个最合适的行维度。
   - 同类重复信息合并；互相矛盾或语义不清的信息用谨慎表述，不强行统一。
   - 无法归入现有维度但重要的信息，优先新增合理维度；只有在用户指定固定模板时，才放入“补充内容”。

4. 填写小结矩阵
   - 每个受访者一列；每个分析维度一行。
   - 单元格内用“提炼结论 + 必要细节 + 少量原话”的方式写；凡是包含多个并列信息、步骤、来源、原因或建议且可以合理拆分的，必须在同一单元格内编号列点，格式为“1. 标签：说明”“2. 标签：说明”。
   - 不要把访谈顺序原样复述；按研究维度重组。
   - 不要把每句话拆成一个点；把相同含义合并成少量要点。
   - 保留足够细节，让后续报告能直接引用或对比。

5. 质量审计
   - 检查每个重要信息块是否进入对应行。
   - 检查每个单元格是否只来自材料。
   - 检查同一行下不同受访者是否可横向比较。
   - 检查是否误把研究员追问、假设或示例当作受访者观点。
   - 检查是否遗漏产品体验问题、未满足需求、机会点、付费意愿等研究价值高的信息。

## 证据边界

- 受访者明确表达的内容可以写成事实，例如“受访者主要用 WPS”。
- 研究员对受访者的复述，只有被受访者确认或上下文明确时，才可纳入。
- 研究员举例、提示、引导语不能直接当作受访者观点。
- 转写不清的内容要保守处理，可写“材料中表述不清，疑似...”或省略。
- 不根据职业、年龄、产品名推断材料里没有的需求。
- 不主动补写市场背景、竞品常识、产品能力或商业判断。

## 写作口径

- 使用中文。
- 使用用户研究员的分析语言：具体、克制、可比较。
- 优先写行为和判断依据，再写态度结论。
- 用“场景：”“流程：”“痛点：”“机会点：”等短标签提高可读性。
- 每个单元格内部通常 2-6 条要点；复杂维度可分小标题。
- 能列点、能总结的内容必须分点输出，不要将多个并列事实压成一段话。例如：
  `1. Zotero：文献阅读器，导入pdf进行阅读或者下载插件自动在网页抓取文献pdf进行阅读`
  `2. AI辅助阅读：遇到公式不理解的情况会问AI，AI可以帮助推导或给予前置数据知识`
- 只有单一事实、单一判断或无法合理拆分的内容才使用一句话；编号应服务于信息区分，不为短句机械拆点。
- 原话只保留最能解释需求、动机或痛点的短句；不要堆长引文。
- 对未提及的内容写“未提及”，不要写“无需求”，除非受访者明确说没有。

## Excel 输出硬规则

默认输出 `.xlsx` 文件，并匹配参考表样式和结构。最终回复只给文件链接和简短说明，不要把完整小结正文重复粘贴到聊天里。

生成 Excel 时必须：

- 至少包含一张小结 sheet。
- 第一列放研究维度或二级问题，后续列放受访者/访谈场次/用户类型。
- 设置自动换行、合理列宽、标题行/标题列样式。
- 冻结首行和首列，方便横向追加和阅读。
- 清理无内容的空 sheet。
- 导出后检查工作簿非空、关键范围内容存在、文本可读。
- 如有条件，渲染预览图进行视觉检查。

只有以下情况可以先输出 Markdown：

- 用户明确要求先看文本预览。
- 当前工具环境无法生成或保存 `.xlsx`。
- 用户只是在讨论 skill 设计，不是在要求产出访谈小结。

即便临时输出 Markdown，也要说明正式交付应生成 Excel，并在可行时继续生成 `.xlsx`。

如果用户提供多个参考小结，先总结参考结构和当前 skill 需要遵守的 schema，再开始处理新笔录；最终仍要生成 Excel。

## 不要做

- 不要默认输出“产品/功能介绍、调研背景、调研目的、访谈问题、访谈小结”的单篇报告，除非用户明确要求这种格式。
- 不要把“补充内容”当作主要归档方式。
- 不要为了覆盖完整而复制大段逐字稿。
- 不要把受访者没有确认的研究员总结写成结论。
- 不要写泛泛洞察，例如“用户注重效率”“体验有待优化”，必须落到具体场景和证据。
