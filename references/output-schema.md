# 中文访谈小结矩阵结构

## 默认交付形态

访谈小结默认交付为 `.xlsx` 文件，并使用矩阵结构：

```text
行 = 研究分析维度
列 = 受访者 / 访谈场次 / 用户类型
单元格 = 该受访者在该维度下的提炼内容
```

示例：

| 维度 | 受访者 A | 受访者 B |
|---|---|---|
| 基本信息 | 年龄、职业、核心场景、常用产品 | 年龄、职业、核心场景、常用产品 |
| 使用场景 | 按场景分类写 | 按场景分类写 |
| 全流程 | 准备、生成/使用、修改/后续行为 | 准备、生成/使用、修改/后续行为 |
| 痛点 | 按产品/流程/场景归类 | 按产品/流程/场景归类 |
| 机会点 | 可优化方向或未满足需求 | 可优化方向或未满足需求 |

如果只有一个受访者，也保留这种“维度-内容”的结构，方便后续横向追加。

不要默认在聊天中输出完整 Markdown 小结。聊天最终回复只提供 `.xlsx` 文件链接、简短说明和必要限制。

## 从参考 Excel 继承结构

当用户提供已有小结表时，按以下优先级继承：

1. sheet 拆分方式，例如“娱乐场景”“学习场景”。
2. 左侧行维度，例如“内容获取”“内容消费”“体验问题”“总结提炼”。
3. 二级行标题，例如“获取方式”“是否转存”“核心需求和关注点”。
4. 列标题命名，例如受访者姓名、编号、职业、用户类型。
5. 单元格内部写法和颗粒度。

不要把已有 Excel 结构改成普通 Markdown。输出 `.xlsx` 时，尽量保持行高、换行、冻结首行/首列、自动换行、宽列阅读等表格可读性。

## 从访谈提纲生成结构

若没有参考 Excel，但有访谈提纲，将提纲转换为分析维度：

- 开场/背景问题 → `基本信息`
- 场景追问 → `使用场景` 或 `典型场景和需求`
- 流程追问 → `用户全流程`、`制作/消费/使用过程`
- 工具或竞品追问 → `工具使用与评价`、`竞品模式认知和评价`
- 痛点追问 → `体验问题` 或 `当前痛点`
- 机会点追问 → `产品机会点`、`可优化方向`
- 付费追问 → `付费动因及付费点排序`
- demo 反馈 → `Demo 效果验证` 或 `产品体验反馈`

不要逐字复制访谈问题做行标题；要把问题组抽象成可比较的研究维度。

## 常见项目 schema

### PPT 生成路径项目

适合行维度：

- 基本信息
- 用户的使用场景（分类写）
- 不同 PPT 使用场景下的需求差异点
- 用户制作 PPT 的全流程
- 不同场景下 PPT 制作的差异点
- 竞品模式选择的认知和评价
- 目标产品三种模式的应用场景
- 目标产品的使用体验问题（可区分模式写）
- AI 生成 PPT 的付费动因及付费点排序
- 目标产品的付费意愿
- 生成后不下载/不用的原因

### 谷子/自制周边项目

适合行维度：

- 基本背景了解
- 整体消费行为
- 买谷子场景/品类/动机/渠道
- 买谷子的产品机会点
- 做谷子场景/品类/动机/渠道
- 做谷子的产品机会点
- 谷子制作过程
- 制作过程中的使用痛点
- 谷子制作后的行为
- 分享、二次加工、打印下单、作品管理等后续行为

### 视频消费/学习项目

可按场景拆 sheet：

- 娱乐场景
- 学习场景

适合行维度：

- 内容获取：知晓触达渠道、获取方式、是否转存、核心需求和关注点
- 内容消费：典型需求、设备和产品、消费习惯
- 播放/学习行为：播放器模式、倍速/清晰度/连播/分屏/笔记等
- AI 相关需求：字幕、文稿、总结、课件、思维导图
- 体验问题
- Demo 效果验证
- 总结提炼：可优化和建设方向

以上只是起点。真实输出必须根据用户给的提纲和材料调整。

## 单元格写法

推荐结构：

```text
1. 场景/行为：具体发生在什么情况下
2. 需求/关注点：用户在意什么，为什么
3. 痛点/评价：哪里不满意，影响是什么
4. 机会点：如果材料支持，可提炼为产品可优化方向
```

可以使用短标签：

```text
场景：
-- ...
需求：
-- ...
痛点：
-- ...
机会点：
-- ...
```

保留原话的方式：

```text
核心需求：希望快速获取完整资源，减少搜索成本。
原话：“若公众号找不到资源，去其他平台搜索会比较麻烦。”
```

原话要短，只用于支持关键判断。不要把长段对话粘进单元格。

## 覆盖规则

整理前做一遍私下覆盖表：

```text
信息块 | 证据来源 | 归入维度 | 是否已写入 | 备注
```

信息块至少包括：

- 用户背景
- 使用/消费/制作场景
- 输入或获取方式
- 工具和产品
- 决策标准
- 行为流程
- 修改/后续行为
- 痛点和不满
- 正向评价
- 未满足需求
- 产品机会点
- 付费意愿或不付费原因
- demo/产品反馈

最终不要展示私下覆盖表，除非用户要求。

## 质量审计

交付前逐项检查：

- 是否沿用了用户提供的参考表结构。
- 是否每个受访者都在同一行维度下可横向比较。
- 是否保留了足够细节，而不是只有抽象结论。
- 是否把研究员问题和受访者回答区分清楚。
- 是否有任何内容来自猜测或外部知识。
- 是否遗漏高价值信息：痛点、动机、场景差异、付费态度、产品机会点。
- 是否把“未提及”和“明确无需求”区分开。
- 是否避免了大段逐字稿堆砌。
- 如果输出 Excel，是否便于阅读：自动换行、合理列宽、标题清楚、空白 sheet 清理。

## Excel 文件要求

生成 Excel 时遵守：

- 文件名包含项目名和受访者/批次，例如 `PPT生成路径访谈小结-李海潮.xlsx`。
- sheet 名使用项目场景或“小结”，例如 `访谈小结`、`娱乐场景`、`学习场景`。
- 第一行是列标题，第一列是维度标题。
- 所有长文本单元格开启自动换行。
- 第一行和第一列冻结。
- 维度列宽约 24-35，受访者内容列宽约 60-100，按内容调整。
- 标题行使用浅色底纹和加粗字体；维度列使用浅色底纹。
- 对填充区域加浅色边框，不要格式化大量空白区域。
- 导出后检查：非空单元格数量、关键行是否存在、没有默认空 sheet。
- 如工具支持，渲染预览图检查可读性；预览图不作为最终交付，除非用户要求。

## Markdown 临时输出

只有用户明确要求预览、不能生成文件，或正在讨论 skill 设计时，才使用 Markdown。临时 Markdown 使用这种结构：

```markdown
# 访谈小结

## 结构说明
- 项目主题：
- 受访者：
- 采用的分析维度：

## 小结矩阵

| 维度 | 受访者 |
|---|---|
| 基本信息 | ... |
| 使用场景 | ... |
| 全流程 | ... |
| 痛点 | ... |
| 机会点 | ... |

## 查漏说明
已覆盖的主要信息包括：...
未写入/未展开的信息：...
```

即使是 Markdown，也要保持矩阵思路。临时预览之后，继续生成正式 `.xlsx`。
